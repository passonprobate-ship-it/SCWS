# SPAWN Project Playbook

Complete operational guide for any AI agent (Claude, OpenCode, or any LLM) working inside SPAWN. This is your cookbook — follow these recipes exactly.

## What Is SPAWN?

SPAWN is a self-programming autonomous web node. You are the AI brain. The server at `/var/www/scws` is your body. You don't suggest code for humans to copy-paste — you write files to disk, build them, start processes, configure routing, and verify everything works. If something breaks, you read logs, diagnose, fix, and redeploy.

## Core Architecture

```
/var/www/scws/
  daemon/              <-- Control plane (Express 5, port 4000, DO NOT RESTART)
    dist/index.cjs     <-- Bundled daemon (single-line JS)
    dist/dashboard.html <-- Web dashboard SPA
    .env               <-- DATABASE_URL, DASHBOARD_TOKEN, etc.
    ecosystem.config.cjs
  projects/            <-- Your creations live here
    <project-name>/    <-- Each project is self-contained
  nginx/projects/      <-- Auto-generated per-project nginx configs
  scripts/             <-- Bootstrap, healthcheck, auto-update
  docs/                <-- This playbook and screenshots
  logs/
```

**You run inside the daemon.** Restarting `scws-daemon` kills your session. Project work NEVER needs a daemon restart. Never run `pm2 restart all`.

## Authentication

All daemon API calls require a Bearer token from `/var/www/scws/daemon/.env` (`DASHBOARD_TOKEN`).

```bash
# Read the token
TOKEN=$(grep DASHBOARD_TOKEN /var/www/scws/daemon/.env | cut -d= -f2)

# Use in curl
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:4000/api/projects
```

## Recipe 1: Create a Project From Scratch

This is the most common operation. Follow ALL steps — skipping any step leaves the project broken or invisible in the dashboard.

### Step 1: Write the code

```bash
mkdir -p /var/www/scws/projects/my-app/src
```

Write your source files. Typical Express project structure:

```
projects/my-app/
  src/index.ts       <-- Express app
  script/build.js    <-- esbuild bundler
  package.json
  tsconfig.json
  .env               <-- PORT, BASE_URL, DATABASE_URL (if needed)
```

**Important patterns for the Express app:**
- Read `PORT` and `BASE_URL` from environment
- All routes must be relative to `BASE_URL` (e.g., `/my-app/api/foo`)
- The app is reverse-proxied by nginx — don't handle SSL
- Listen on `0.0.0.0`, not `127.0.0.1`

Example minimal Express app (`src/index.ts`):

```typescript
import express from "express";
const app = express();
const PORT = parseInt(process.env.PORT || "5050");
const BASE = process.env.BASE_URL || "/my-app";

app.get(`${BASE}/`, (req, res) => res.json({ status: "ok" }));
app.get(`${BASE}/health`, (req, res) => res.json({ healthy: true }));

app.listen(PORT, "0.0.0.0", () => {
  console.log(`my-app listening on port ${PORT}`);
});
```

Example esbuild script (`script/build.js`):

```javascript
const { build } = require("esbuild");
build({
  entryPoints: ["src/index.ts"],
  bundle: true,
  platform: "node",
  target: "node20",
  outfile: "dist/index.js",
  format: "cjs",
  sourcemap: true,
  external: ["pg-native"],
}).then(() => console.log("Build complete"));
```

### Step 2: Install dependencies

```bash
cd /var/www/scws/projects/my-app
npm init -y
npm install express
npm install -D typescript esbuild @types/express @types/node
```

### Step 3: Create the .env file

```bash
cat > /var/www/scws/projects/my-app/.env << 'EOF'
PORT=5050
BASE_URL=/my-app
NODE_ENV=production
EOF
```

The port must be in the 5001-5099 range and not conflict with existing projects. Check existing ports:

```bash
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:4000/api/projects | jq '.[].port'
```

### Step 4: Build

```bash
cd /var/www/scws/projects/my-app
node script/build.js
```

### Step 5: Start with PM2

```bash
pm2 start /var/www/scws/projects/my-app/dist/index.js \
  --name my-app \
  --cwd /var/www/scws/projects/my-app \
  --node-args="--env-file=.env --max-old-space-size=128" \
  --max-memory-restart 150M

pm2 save
```

**Heap cap guidelines:**
| Project Size | Heap Cap | PM2 Restart |
|---|---|---|
| Tiny (static, landing) | 64 MB | 100M |
| Small (simple API) | 128 MB | 150M |
| Medium (DB + API) | 192 MB | 200M |
| Large (heavy processing) | 256 MB | 300M |

### Step 6: Create nginx config

Write a location block to `/var/www/scws/nginx/projects/my-app.conf`:

```nginx
# Project: my-app (port 5050)
# Auto-generated by SPAWN daemon -- do not edit manually
location /my-app/ {
    proxy_pass http://127.0.0.1:5050/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection 'upgrade';
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_set_header X-Base-Path /my-app;
    proxy_read_timeout 300s;
    proxy_connect_timeout 10s;
    proxy_intercept_errors on;
    error_page 502 503 504 = @project_down;
}
```

Then reload nginx:

```bash
sudo nginx -t && sudo nginx -s reload
```

### Step 7: Register the project card in the daemon

This makes the project visible in the SPAWN dashboard. **This is the step most agents miss.**

```bash
TOKEN=$(grep DASHBOARD_TOKEN /var/www/scws/daemon/.env | cut -d= -f2)

# CREATE the project card
curl -s -X POST http://localhost:4000/api/projects \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my-app",
    "displayName": "My App",
    "framework": "express",
    "description": "Short description of what this project does",
    "needsDb": false
  }'
```

**POST /api/projects fields:**

| Field | Required | Type | Notes |
|---|---|---|---|
| `name` | Yes | string | Lowercase, alphanumeric + hyphens. Must match directory name. |
| `displayName` | Yes | string | Human-readable name shown on card. |
| `framework` | No | string | `express`, `next`, `static`, `bash`, etc. Default: `express` |
| `description` | No | string | One-line description. |
| `needsDb` | No | boolean | If true, daemon auto-creates a PostgreSQL database. |
| `gitRepo` | No | string | GitHub URL (e.g., `https://github.com/user/repo`). |

The daemon auto-assigns the next available port. The response returns the full project object with `id` and `port`.

**IMPORTANT:** The daemon assigns its own port. If you already chose a port for your .env/PM2, you should PATCH the project to match, or use the port the daemon assigned.

Then PATCH to set status and wire up features:

```bash
# Set status to running + add metadata
curl -s -X PATCH http://localhost:4000/api/projects/my-app \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "status": "running",
    "description": "My cool project description",
    "port": 5050,
    "gitRepo": "https://github.com/user/repo",
    "gitBranch": "main",
    "deployTargets": "[{\"name\":\"prod-vps\",\"host\":\"1.2.3.4\",\"user\":\"spawn\",\"path\":\"/var/www/scws/projects/my-app\",\"method\":\"scp\"}]"
  }'
```

**PATCH /api/projects/:name fields** (all optional):

| Field | Type | Notes |
|---|---|---|
| `status` | string | `running`, `stopped`, `building`, `error` |
| `displayName` | string | Update display name |
| `description` | string | Update description |
| `port` | integer | Must be unique in 5001-5099 range |
| `framework` | string | `express`, `next`, `static`, `bash` |
| `gitRepo` | string | GitHub URL -- enables GitHub tab on project detail |
| `gitBranch` | string | Branch name -- shown in GitHub tab |
| `dbName` | string | Database name -- shown on card stats |
| `entryFile` | string | Default: `dist/index.js` |
| `buildCommand` | string | Custom build command |
| `startCommand` | string | Custom start command |
| `deployTargets` | string | JSON array of deploy target objects (stringified) |

**Deploy target object shape:**
```json
{
  "name": "prod-vps",
  "host": "1.2.3.4",
  "user": "spawn",
  "path": "/var/www/scws/projects/my-app",
  "method": "scp"
}
```

## Recipe 2: Create a Database for a Project

```bash
# Create the database
sudo -u postgres psql -c "CREATE DATABASE my_app_db OWNER scws;"

# Add to project .env
echo "DATABASE_URL=postgresql://scws:$(grep SCWS_DB_PASSWORD /var/www/scws/daemon/.env | cut -d= -f2)@localhost:5432/my_app_db" >> /var/www/scws/projects/my-app/.env

# Update the project card so the DB name shows on the dashboard
curl -s -X PATCH http://localhost:4000/api/projects/my-app \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"dbName": "my_app_db"}'

# Rebuild + restart to pick up new env var
cd /var/www/scws/projects/my-app && node script/build.js
pm2 restart my-app
```

The DB password for the `scws` PostgreSQL user is in `/var/www/scws/daemon/.env` as `SCWS_DB_PASSWORD`.

## Recipe 3: Enable Tailscale Funnel (Public Internet Access)

Tailscale Funnel exposes one project at a time to the public internet via HTTPS.

**Prerequisites:**
- Tailscale installed and authenticated (`tailscale status` should show this node)
- Funnel enabled on your tailnet at https://login.tailscale.com/admin/dns (under "HTTPS Certificates" and "Funnel")

### Via Dashboard API

```bash
# Enable funnel for a running project
curl -s -X POST http://localhost:4000/api/funnel/enable \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"projectName": "my-app"}'

# Check funnel status
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:4000/api/funnel/status | jq .

# Disable funnel
curl -s -X POST http://localhost:4000/api/funnel/disable \
  -H "Authorization: Bearer $TOKEN"
```

### What the API does internally

1. Resets any existing funnel: `sudo tailscale funnel reset`
2. Enables funnel on the project's port: `sudo tailscale funnel --bg <port>`
3. Saves the active project to `daemon_config` table (key: `funnel-active-project`)
4. The project becomes accessible at `https://<hostname>.tail<tailnet>.ts.net`

### Via CLI (manual)

```bash
# Enable
sudo tailscale funnel --bg 5050

# Check status
sudo tailscale funnel status --json

# Disable
sudo tailscale funnel reset
```

**Important:** Only ONE project can use Funnel at a time. The daemon's Funnel page in the dashboard provides a UI for switching between projects.

## Recipe 5: Delete a Project

```bash
# Stop PM2 process
pm2 stop my-app
pm2 delete my-app
pm2 save

# Remove nginx config
rm /var/www/scws/nginx/projects/my-app.conf
sudo nginx -t && sudo nginx -s reload

# Delete from daemon registry
curl -s -X DELETE http://localhost:4000/api/projects/my-app \
  -H "Authorization: Bearer $TOKEN"

# Remove files
rm -rf /var/www/scws/projects/my-app

# Drop database if one was created
sudo -u postgres psql -c "DROP DATABASE IF EXISTS my_app_db;"
```

## Recipe 6: Deploy a Project to a Remote VPS

Projects with `deployTargets` configured can be deployed via the dashboard or API.

```bash
curl -s -X POST http://localhost:4000/api/projects/my-app/deploy \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"targetName": "prod-vps"}'
```

The daemon will: build locally, SCP the dist bundle + package.json to the target, run `npm install --omit=dev` remotely, and `pm2 restart` on the target.

## nginx Gotchas: Trailing Slashes and Base URL

**The trailing slash on `proxy_pass` matters.** With `location /my-app/` and `proxy_pass http://127.0.0.1:5050/;` (note the trailing `/`), nginx strips the location prefix. A request to `/my-app/api/health` arrives at your app as `/api/health`.

**Your app must use `BASE_URL` for self-referencing URLs only** (links, redirects, static assets), NOT for route definitions. Define routes without the prefix:

```javascript
// CORRECT — routes are prefix-free thanks to nginx stripping
app.get("/api/health", (req, res) => res.json({ ok: true }));
app.use("/static", express.static("public"));

// WRONG — don't duplicate the prefix in routes
app.get("/my-app/api/health", ...);  // This won't match!
```

**Common mistakes:**
- Missing trailing `/` on `proxy_pass` → requests arrive with `/my-app/...` prefix intact
- Hardcoding `/my-app/` in route definitions → double-prefixed, nothing matches
- Forgetting `BASE_URL` in HTML links → links point to `/` instead of `/my-app/`
- Not setting `BASE_URL=/<name>` in `.env` → app can't construct correct self-referencing URLs

**Testing tip:** Always test through nginx first (`curl http://localhost/<name>/`), not just the direct port.

## Dashboard HTML Editing Guide

The dashboard is a single-file SPA at `daemon/dist/dashboard.html` (7200+ lines, vanilla JS). The daemon is a bundled file at `daemon/dist/index.cjs` (single line).

**Editing dashboard.html** is straightforward — it's readable HTML/CSS/JS. Search for function names or HTML patterns.

**Editing index.cjs** is harder because it's minified into one line. Key variable names in the bundle:
- `$` = Express app instance
- `D(label, handler)` = async route wrapper with logging
- `Y(req, "name")` = extract route param (like `req.params.name`)
- `w` = storage object (all DB queries: `w.getProject()`, `w.getProjects()`, `w.updateProject()`, `w.logActivity()`, etc.)
- `Ws` = DASHBOARD_TOKEN value
- `Tc(provided, expected)` = timing-safe token comparison

**Route pattern:**
```javascript
$.post("/api/new-route", D("Description", async (t, e) => {
  // t = req, e = res
  let name = Y(t, "name"); // extract :name param
  // ... your logic ...
  e.json({ ok: true });
}));
```

**CRITICAL:** Any changes to `index.cjs` or `dashboard.html` require a daemon restart to take effect. Batch all changes, warn the user, restart as the final command.

## Memory Tools (spawn-mcp)

SPAWN has a persistent key-value memory system via the `spawn-mcp` MCP server. Use it to save task plans, architecture decisions, and state that should survive session disconnects.

**Available MCP tools:**
- `spawn_remember(key, value, tags?)` — Store/update a memory (upserts)
- `spawn_recall(key)` — Retrieve a memory by key
- `spawn_forget(key)` — Delete a memory
- `spawn_list_memories(tag?)` — List all keys (optionally filter by tag)

**Best practice:** Before starting non-trivial work, save your plan:
```
spawn_remember("active-task-my-app", "Step 1: ..., Step 2: ..., Status: starting")
```
Update after milestones. Mark complete or paused when done.

## Conventions and Rules

1. **TypeScript**: `camelCase` variables/functions, `PascalCase` types
2. **Express routes**: wrap with `asyncHandler()` (or `D()` in the bundle)
3. **DB queries**: always through `storage.*` methods, never raw SQL in routes
4. **Shell commands**: use `child_process.execFile` (not `exec`) for safety
5. **Dashboard**: vanilla JS, `api()` helper for authenticated fetch
6. **Paths**: always absolute (`/var/www/scws/...`), never relative
7. **Ports**: 4000 is daemon (fixed), 5001-5099 for projects
8. **Memory**: Pi has 8GB RAM. Keep builds lean. Use heap caps on PM2 processes.

## Troubleshooting

```bash
# Project not accessible via nginx
sudo nginx -t                          # Check config syntax
cat /var/www/scws/nginx/projects/*.conf # Review all configs
sudo nginx -s reload                   # Reload after fixes

# Project crashing
pm2 logs my-app --lines 50            # Check logs
pm2 show my-app                       # Check restarts, memory, uptime

# Project card missing from dashboard
curl -s -H "Authorization: Bearer $TOKEN" http://localhost:4000/api/projects | jq '.[].name'
# If missing, create it (Recipe 1, Step 7)

# Database connection failing
sudo -u postgres psql -l              # List all databases
sudo -u postgres psql scws_daemon -c "SELECT name, port, status FROM projects;"

# Port conflict
pm2 jlist | jq '.[].pm2_env.PORT'    # Check PM2 ports
ss -tlnp | grep '50[0-9][0-9]'       # Check listening ports
```
